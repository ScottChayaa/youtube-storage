# YouTube 回憶備份與檢索系統 — 需求分析／設計文件

- **日期**：2026-07-05
- **狀態**：設計定案（待轉為實作計畫）
- **一句話**：把個人照片／影片上傳到自己的 Google 生態（YouTube 存媒體、Drive 存索引）做**異地備份**，並提供一個能用 tag／關鍵字**秒跳到某影片某片段**的檢索網站。

---

## 1. 背景與目標

### 動機
使用者每月都會拍照與拍短片，希望：
1. 有一個**免費**的異地備份，防止本機電腦損毀導致回憶全失。
2. 不必一支一支影片慢慢找，能用關鍵字快速定位到「某個回憶片段」。
3. 未來能把同一套系統分享給親友，各自用自己的 Google 帳號備份與檢索，互不干擾。

### 核心目標
- **(G1) 異地備份**：把回憶上傳到使用者**自己的** Google 空間（YouTube 存媒體、Drive 存索引），防本機損毀。
- **(G2) 快速檢索**：用 tag／關鍵字，秒跳到某支影片的某個時間片段（`youtube.com/watch?v=xxx&t=185s`）。
- **(G3) 各自獨立、可分享**：架構天生多人，親友可用自己的 Google 帳號各玩各的，資料互不相干。

### 非目標（YAGNI）
- 第一版不做 AI 影像辨識／人臉辨識（架構預留，日後追加）。
- 不做原生 App（改用 PWA 網站）。
- 第一版不公開上架、不走 Google 正式驗證（朋友以「測試者」模式使用）。
- 不在既有 YouTube 影片上「插入／編輯」內容（YouTube 不支援，改用 append 模型，見 §7）。
- 第一版只實作 YouTube 單一儲存後端（介面已抽象化，其他平台日後可插拔加入，見 §3.1）。

## 2. 名詞定義

| 名詞 | 意義 |
|------|------|
| **event（事件）** | 一個虛擬的邏輯集合，例如「2025-07 游泳」。**可橫跨多支影片、可隨時間陸續補充**。 |
| **video（影片）** | 一支實體 YouTube 影片。型別為「投影片影片（照片合成）」或「短片（原樣上傳）」。 |
| **segment（片段）** | **搜尋的最小單位**。對應影片中的一段時間（起訖秒數）與一個來源檔（一張照片或一段短片）。 |

---

## 3. 系統架構總覽

全部圍繞 Google 生態，**無中央伺服器、無資料庫月租、無儲存成本**。每位使用者的資料都在自己帳號內。

```
┌─────────────────────┐   上傳(不公開)   ┌──────────────────────┐
│  本機桌面整理工具    │ ───────────────▶ │  YouTube（媒體倉庫）  │
│  (掃描/標籤/合併/上傳)│                  └──────────────────────┘
└──────────┬──────────┘
           │ 產生索引 JSON + 縮圖
           ▼
┌──────────────────────┐
│  使用者的 Google Drive │ ◀───── 讀取索引 ─────┐
│  (App 專屬隱藏資料夾)  │                       │
└──────────────────────┘                       │
                                    ┌───────────┴───────────┐
                          Google 登入│  檢索 PWA 網站         │
                                    │ (手機/電腦, 瀏覽器內搜尋)│
                                    └───────────────────────┘
```

三個 Google 服務：**Google 登入（OAuth）／ YouTube Data API ／ Drive API**。

### 3.1 可插拔儲存後端（延展性預留）

系統把「索引」與「媒體倉庫」解耦，因此媒體平台是可替換的。定義一個**儲存後端介面**，每個後端實作三件事：

| 方法 | 說明 |
|------|------|
| `upload(media)` | 上傳影片到該平台，回傳 `platform_video_id` |
| `deep_link(video_id, sec)` | 產生跳到指定秒數的觀看連結 |
| `download(video_id)` | 還原時下載回檔案 |

索引的 video / segment 帶 `platform` 欄位以識別來源（見 §6）。

**Phase 1 只實作 `youtube` 一個後端。** 經評估（2026-07-05）：市面上沒有第二個平台同時滿足「無限 + 免費 + 永久 + 私密 + 時間跳轉」——Vimeo／Dailymotion 有容量上限、Twitch 會自動刪除 VOD、Bilibili 需實名與審核且隱私不足、Internet Archive 無限免費但全公開。故現階段 **YouTube 為唯一可行後端**；抽象層僅為未來延展預留，不主動新增其他平台。

## 4. 元件一：本機桌面整理工具

在使用者電腦上執行（需要 ffmpeg 做影片處理）。流程：

1. **掃描** `年-月` 資料夾（使用者已按年月分好）。
2. **自動抽 metadata 打底 tag**：
   - 年、月、日 → 從 EXIF／檔名
   - 地點 → 從 GPS 反查（若照片有定位）
3. **時間軸編輯器**（像影片剪輯軟體的體驗）：
   - 排列照片／短片
   - 框選區間、把某段的照片影片集中在一起
   - 集中補標籤、修改標籤
   - 旁邊預覽視窗，時間戳記可跳轉
4. **合併（採 A 案，見 §7）**：
   - 照片 → 合成一支**投影片影片**（高解析度編碼，見 §10）
   - 短片 → **原樣各自上傳**（不重新編碼混合）
5. **上傳 YouTube（設為不公開）**，並把章節寫進影片描述（YouTube 原生跳轉 + 索引備份）。
6. **產生索引檔（JSON）+ 每個片段的縮圖**，存到使用者自己的 Google Drive。
7. （還原輔助）保留「哪支影片來自哪批原檔」的對應，讓日後從 YouTube 下載還原有意義。

---

## 5. 元件二：檢索 PWA 網站

Google 登入後，從使用者**自己的 Drive** 讀取索引，全部在瀏覽器內搜尋（client-side，無後端）。

### 主動檢索
- 輸入一段文字／幾個關鍵字（例：`2025 游泳 家人`）
- 比對 segments 的 tags／說明 → 排序 → 列出符合的影片與一串**可點的時間點**
- 點擊 → 直接跳 `https://youtube.com/watch?v={id}&t={sec}s`

### 被動瀏覽
- 購物網站式首頁：分類好的**縮圖牆**、最近更新、片段預覽
- 看到某片段可得知「大概做了什麼、有哪些關鍵字」
- 發現缺漏的 tag → 可回頭補充（回到整理工具補標）

### 存取控制
- 需 Google 登入才能使用；每人只讀得到自己 Drive 的索引。

## 6. 索引資料模型（系統的心臟）

- **格式**：JSON（瀏覽器原生解析、零依賴、快）。工具可額外匯出 YAML 快照供肉眼查看，但網站正式載入 JSON。
- **位置**：使用者 Google Drive 的 **App 專屬隱藏資料夾（appDataFolder）**。

### 結構範例

```json
{
  "version": 1,
  "updated_at": "2026-07-05T10:00:00Z",
  "events": [
    {
      "id": "evt_2025_07_swim",
      "name": "2025-07 游泳",
      "year": 2025,
      "month": 7,
      "location": "宜蘭"
    }
  ],
  "videos": [
    {
      "id": "vid_abc123",
      "platform": "youtube",
      "platform_video_id": "abc123",
      "type": "slideshow",
      "title": "2025-07 游泳 照片",
      "duration_sec": 120,
      "upload_date": "2026-07-05",
      "source_batch": "2025-07/游泳/"
    }
  ],
  "segments": [
    {
      "id": "seg_0001",
      "event_id": "evt_2025_07_swim",
      "platform": "youtube",
      "platform_video_id": "abc123",
      "start_sec": 180,
      "end_sec": 184,
      "source_file": "2025-07/游泳/IMG_1234.jpg",
      "thumbnail": "thumbs/seg_0001.jpg",
      "tags": ["2025", "7月", "游泳", "宜蘭"],
      "caption": "第一次下水",
      "date": "2025-07-12",
      "gps": {"lat": 24.70, "lng": 121.74}
    }
  ]
}
```

- 投影片影片：每張照片 = 一個 segment（起訖秒數落在該影片內）。
- 短片：整段 = 一個 segment（未來可手動細切成多個）。
- 每個 video / segment 帶 `platform` 欄位（目前一律 `youtube`）；跳轉連結由各後端的 `deep_link()` 產生（見 §3.1）。
- 搜尋 = 關鍵字比對 segments 的 `tags` / `caption`，回傳「縮圖 + 跳轉連結」。

## 7. 資料流

### 合併策略（A 案）
一個事件裡：**照片合成投影片影片；原始短片原樣各自上傳（不重新編碼混合）**。
- 好處：短片畫質損失最少（只經 YouTube 壓一次）、處理快、模型單純。
- 「檔案較多」對使用者透明——都透過檢索網站進入，索引會把零散影片隱藏成「片段」。

### 補資料（append-only 模型）
YouTube 影片**無法插入／追加**內容（只能整支刪掉重傳、會換掉網址）。因此補資料的做法是：

> **開一支新的「補充影片」上傳，並在索引把它的片段標到同一個 event。**

搜尋時查索引，新舊片段自動一起出現，使用者感覺不到它們是不同影片。**舊資料永遠不用重做。**

核心原則：**實體 YouTube 影片是「只能往後追加」的容器；event 是索引裡的虛擬集合，可由多支影片、在不同時間陸續拼湊而成。**

### 還原（從備份取回）
本機損毀時，可透過索引得知「哪支 YouTube 影片對應哪批原檔」，從 YouTube 下載回來。工具提供「下載回自己影片」的還原輔助。**注意：取回的是壓縮版，非原始畫質（見 §10）。**

## 8. Tag 策略（分階段）

| tag 類型 | 例子 | 來源 | 階段 |
|---------|------|------|------|
| 時間 | `2025`、`7月` | EXIF／檔名（自動） | Phase 1 |
| 地點 | `宜蘭`、`海邊` | GPS 反查（自動，若有定位） | Phase 1 |
| 內容／事件 | `游泳`、`家人` | 手動標記（時間軸編輯器） | Phase 1 |
| 內容（自動） | `游泳`、`海邊`、`蛋糕` | AI 影像辨識 | Phase 3 |
| 人物 | `家人某某` | AI 人臉辨識 | Phase 3（之後） |

Phase 1 方針：**metadata 自動打底（時間／地點）＋ 強大的手動標記工具**。AI 因為索引與打標分層，日後只是「多一個往索引補 tag 的來源」，不動既有架構。

## 9. 多使用者與 Google OAuth／驗證

- **模型**：每位使用者用自己的 Google 帳號登入，授權 YouTube（讀取／上傳）＋ Drive（存索引）。資料各自獨立、互不相干。
- **權限範圍**：YouTube 與 Drive 屬 Google 的「敏感／受限」範圍。
- **第一版（自己 + 親友）**：App 設在「測試中」狀態，把親友加入**測試者名單（≤100 人）**即可使用；登入時會出現「未驗證 App」警告畫面，按繼續即可。**免費、不用送審。**
- **公開（未來里程碑）**：若要開放給不特定多數人，需走 Google 正式驗證（受限範圍可能需第三方安全稽核，耗時／花錢）。列為 Phase 4。

## 10. 備份定位與關鍵限制（誠實聲明）

**YouTube 是「有損的異地備份」。** YouTube 一定會重新壓縮，從 YouTube 下載回來的是壓縮版，**不是原始畫質**：

- **短片**：壓一次，救回來畫質尚可（堪用）。
- **照片（合併成投影片）**：最弱——從影片抓某一格還原，得到的是壓縮過的影片畫格，**不是原本那張 JPEG**，細節損失明顯。

**設計對策：**
- (a) 投影片採**高解析度（如 4K）編碼**上傳，壓縮後盡量保留細節。
- (b) 建議（非強制）：照片對使用者重要時，**至少再留一份真原檔**（便宜外接硬碟，或原始 JPEG 直接放 Google Drive——Drive 是檔案儲存不會壓縮，15GB 免費額度內）。這與 YouTube 備份不衝突，是雙保險。

**定位總結**：YouTube 這層備份保證「不會全部歸零、能救回堪用的一份」，價值實在；但**不是原檔等級的忠實備份**，尤其對照片。

## 11. 風險與限制彙整

- YouTube／Drive 上的媒體均為**壓縮後畫質**，無損版在本機（見 §10）。
- 「不公開」≠ 加密；索引集中了所有連結，故檢索網站以 Google 登入保護。
- 有版權配樂的影片可能被 YouTube Content ID 靜音／封鎖（即使不公開）。
- 依賴 Google／YouTube 的免費政策，未來政策可能改變。
- Google Drive 免費額度 15GB（與 Gmail／Photos 共用）；索引檔極小（數百 KB），縮圖數十 MB 等級，不成問題。
- YouTube 未驗證帳號單支影片上限 15 分鐘；已做過手機驗證的帳號可到 12 小時／256GB（使用者已符合）。

## 12. 分階段藍圖

- **Phase 1（MVP，先給自己用）**
  - 桌面工具：掃描 → metadata 自動 tag → 時間軸手動標記 → A 案合併 → 上傳 YouTube（不公開）→ 產生索引 JSON + 縮圖到 Drive。
  - 檢索 PWA：Google 登入 → 主動搜尋（關鍵字→影片+時間點）→ 基本縮圖牆。
  - 明確達成 (G1) 異地備份 + (G2) 檢索。索引記錄「影片↔原檔批次」對應，讓還原有意義。
- **Phase 2**：被動瀏覽強化（片段預覽、最近更新、補標流程）；開放親友（測試者模式）。
- **Phase 3**：AI 影像內容 tag → 之後人臉辨識。
- **Phase 4（可選）**：Google 正式驗證、公開給不特定多數人。
